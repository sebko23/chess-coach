## 2. Recent Wins (Resolved This Cycle)

### 2.1 Narration pipeline — end-to-end LLM coaching

**What was broken:** The Coach panel produced silent templates or "Backend not found" errors instead of LLM-generated move explanations.

**Root causes (7 bugs, across 2 commits):**
- LLM pipeline not wired through to the Coach panel
- Wrong OpenRouter model (paid) selected → switched to free tier
- Groundedness flag inverted — `grounded: true` was reported for template fallback
- Narration service swallowed upstream errors → user saw empty bubble
- PV-line moves occasionally `null` from engine → null-deref crash
- CoachPanel sent the wrong field shape to the backend (not the backend's fault)
- Practice deck was missing a `TreeStateProvider` → store crash on tab switch

**Resolution commits:** `bf03eee`, `1c171be`, `0f81bc2`, `9fe088a`, `919d441`, `17d3ee6`, `aef9cca`

**Result:** CoachPanel now narrates every played move via OpenRouter free model; Practice tab loads cleanly; blunder review feedback reaches the LLM.

### 2.2 Maia-1500 (lc0) human-like engine integration

**What was broken:** lc0 was registered as an engine but `analyze` calls returned no PV, so the "human-like" use-case (training against opponent modeling) was non-functional.

**Root causes (5 bugs):**
- Backend UCI bridge didn't pass `--weights` to lc0
- Backend tried `Info` line fields not emitted by lc0
- lc0's required backend flags (`--backend`, `--threads`) missing in the engine spec
- Engine output buffering race vs Go-side read loop
- Maia weights path was relative to backend CWD, not the absolute Tauri data dir

**Resolution commit:** `5c05764`

**Result:** Maia-1500 (lc0) now produces valid PV lines and Elo-bucketed move priors. Documented in `SESSION-HANDOVER-MAIA-FIX.md`.

### 2.3 Tauri `appDataDir` mismatch — EnginesPage "no engines installed"

**What was broken:** `EnginesPage → Local` tab rendered empty even though `engines.json` existed on disk, was valid JSON, passed the zod schema, and had correct permissions.

**Root cause:** `engines.json` is **frontend-owned** (read/written by the Tauri webview via `appDataDir()`), so it lives under `~/.local/share/org.encroissant.app/engines/` (because `tauri.conf.json` → `identifier: org.encroissant.app`), **not** under the backend's `~/.local/share/chess-coach/engines/`. Populate scripts had been writing to the backend path.

**Resolution commit:** `cfd6603`

**Result:** Engines list now populates with both Stockfish 18 (ELO 3500) and Maia-1500 (lc0, ELO 1500); diagnostic debug-channel write removed from `apps/desktop/src/state/atoms.ts`. Documented in `SESSION-HANDOVER-APP-DATA-DIR-FIX.md` (154 lines, includes a "do not fix this list" of files that correctly use `chess-coach` for backend-owned resources).

## 3. Problems Encountered & Lessons Learned

### 3.1 Wrong assumption across multiple sessions

The `chess-coach` vs `org.encroissant.app` confusion spanned at least **three sessions** before it was identified. Root cause: a single project had two distinct data-directory conventions and they shared a similar layout. The investigation followed the wrong tree for so long because:

1. The file name (`engines.json`) matched the backend pattern.
2. JSON validity, zod schema, and file permissions all checked out.
3. An earlier investigation fabricated an internally-consistent wrong story (`enginesListAtom`, `read_engines_file`, `EngineListResponse`) that never existed in the source.

**Lesson captured in handover §7:** "everything checks out applied to a problem is a strong signal that the thing being checked is not the thing that's actually wrong."

**Lesson captured in handover §6:** any new Tauri-webview-owned file must be cross-checked against `tauri.conf.json` → `identifier`, **never** assumed from project name. Concrete shell check documented for future agents.

### 3.2 OpenRouter free-model substitution

Original LLM choice was a paid OpenRouter model. When the API key scope was clarified, we had to swap mid-pipeline. The fix touched:
- `services/chess_coach/llm_router/`
- `apps/desktop/src/state/atoms.ts` (CoachPanel atom)
- `.a0proj/plugins/_model_config/config.json`

The pipeline was already correctly abstracted, so the swap was a single point of change — a good return on the early router abstraction.

### 3.3 Stale webview module cache (small, open)

Even after the `enginesFileStorage.getItem` patch landed and `pnpm tsc --noEmit` was clean, the diagnostic write still did not appear in the `engines/` directory. The patched module is on disk and type-checks, but the running webview is loading some other version of it. This is acknowledged in `SESSION-HANDOVER-APP-DATA-DIR-FIX.md §7` as a follow-up and does **not** affect the path-mismatch diagnosis.

### 3.4 pnpm lockfile format drift

`f4417b8` regenerated `pnpm-lock.yaml` after the lockfile format changed under pnpm v10+. CI must be pinned to a single pnpm version going forward (tracked informally; not yet an ADR).
